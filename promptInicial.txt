Você está trabalhando no workspace GitHub do projeto “Nossa Grana”.

Sua missão é analisar o repositório e implementar um aplicativo de gestão financeira compartilhada para casais e famílias, seguindo DDD, Clean Architecture, Clean Code e SOLID.

O projeto deve utilizar um único código-fonte Flutter, gerando dois artefatos:

Não entregue apenas recomendações ou exemplos. Inspecione o workspace, crie um plano e comece a implementação incremental, mantendo o repositório compilável e testável ao final de cada etapa.

==================================================
2. VISÃO DO PRODUTO
==================================================

Nome provisório: Nossa Grana.

O Nossa Grana funciona como uma planilha financeira compartilhada para casais e famílias.

Um usuário cria um espaço financeiro, como “Finanças da Família”, e convida outra pessoa.

Os membros administram juntos:

- Cartões de crédito.
- Titulares dos cartões.
- Limites.
- Compras à vista.
- Compras parceladas.
- Parcelas atuais e futuras.
- Faturas.
- Pagamentos totais e parciais.
- Próximos vencimentos.
- Empréstimos.
- Parcelas de empréstimos.
- Categorias.
- Lembretes.
- Histórico de atividades.
- Permissões dos membros.

Exemplo principal:

- Guilherme utiliza o aplicativo Android nativo.
- Mariana utiliza o Web/PWA no iPhone.
- Ambos entram no mesmo espaço “Finanças da Família”.
- Guilherme registra uma compra pelo Android.
- Mariana visualiza a compra no Web.
- Mariana registra um pagamento.
- Guilherme visualiza a fatura atualizada no Android.

O sistema deve identificar quem realizou cada ação.

Exemplos:

- “Guilherme adicionou uma compra.”
- “Mariana registrou o pagamento da fatura.”
- “Compra cadastrada por Mariana.”
- “Cartão pertencente a Guilherme.”

==================================================
3. OBJETIVOS DA EXPERIÊNCIA
==================================================

O usuário deve conseguir:

1. Descobrir em menos de 5 segundos quanto ainda pode gastar.
2. Registrar uma compra em menos de 20 segundos.
3. Visualizar quanto precisa pagar no mês.
4. Identificar valores pagos, pendentes e vencidos.
5. Entender o impacto das compras parceladas.
6. Visualizar os próximos meses comprometidos.
7. Identificar os próximos vencimentos.
8. Saber quem cadastrou cada informação.
9. Compartilhar os dados com outro usuário de forma segura.
10. Usar o aplicativo sem conhecimento avançado de finanças.

Priorizar:

- Limite disponível.
- Limite comprometido.
- Faturas atuais.
- Valores pendentes.
- Próximos vencimentos.
- Ações rápidas.
- Clareza de parcelamentos.
- Colaboração entre os membros.

Evitar:

- Excesso de gráficos.
- Linguagem técnica.
- Aparência de planilha tradicional.
- Interface bancária complexa.
- Gamificação excessiva.
- Informações demais na mesma tela.

12. MODELO DE BANCO DE DADOS
==================================================

Implemente todas as tabelas abaixo no schema do SQL Connect/PostgreSQL.

Use UUID como chave primária, salvo justificativa técnica.

Todas as tabelas compartilhadas devem possuir `financial_space_id`.

Utilize constraints, foreign keys, índices, versionamento otimista e unicidade onde necessário.

----------------------------------------------
TABELA 1 — user_profile
----------------------------------------------

Colunas:

- id UUID PK.
- firebase_uid VARCHAR NOT NULL UNIQUE.
- email VARCHAR NOT NULL.
- normalized_email VARCHAR NOT NULL.
- display_name VARCHAR NOT NULL.
- photo_url VARCHAR NULL.
- locale VARCHAR NOT NULL DEFAULT 'pt_BR'.
- timezone VARCHAR NOT NULL DEFAULT 'America/Sao_Paulo'.
- created_at TIMESTAMPTZ NOT NULL.
- updated_at TIMESTAMPTZ NOT NULL.
- last_seen_at TIMESTAMPTZ NULL.

Regras:

- Nunca armazenar senha.
- `firebase_uid` vincula ao Firebase Authentication.
- `normalized_email` deve ser usado para comparação.

Índices:

- unique firebase_uid.
- index normalized_email.
